# Техническое задание (ТЗ)
## Проект: «Кванториум 22» — образовательная LMS-платформа

## 1. Цель документа
Определить требования к разработке полноценной веб-платформы вместо текущего статического сайта:
- backend на **Django (Python)**,
- frontend на **HTMX + Alpine.js + JS/CSS**,
- БД **PostgreSQL**,
- контейнеризация через **Docker**,
- работа в **Git**.

Документ учитывает новые требования: преподаватель должен создавать курсы по единому удобному шаблону, добавлять уроки, изображения, код с подсветкой синтаксиса, практические задания с автоматической проверкой и видеоматериалы.

---

## 2. Контекст текущего состояния
Сейчас проект — набор статических страниц (главная, достижения, новости, учебные HTML-уроки). Необходимо перейти к динамической архитектуре, где весь контент хранится в БД и управляется через внутренние интерфейсы.

---

## 3. Бизнес-цели
1. Создать централизованную систему управления образовательным контентом.
2. Обеспечить регистрацию пользователей и назначение ролей администратором.
3. Дать преподавателю удобный конструктор курса и уроков по шаблону.
4. Реализовать админ-панель для CRUD-операций: курсы, уроки, новости, достижения, отзывы, пользователи.
5. Поддержать практические задания с проверкой.
6. Реализовать базовый мониторинг состояния сервиса и аудит действий.

---

## 4. Технологический стек
- **Backend:** Python 3.12+, Django 5.x
- **DB:** PostgreSQL 15+
- **Frontend:** Django Templates + HTMX + Alpine.js + vanilla JS + CSS
- **WSGI/ASGI:** Gunicorn (или Uvicorn)
- **Reverse proxy:** Nginx
- **Инфраструктура:** Docker, Docker Compose
- **Хранение файлов:** local media (MVP), S3-совместимое хранилище (этап масштабирования)
- **Подсветка кода:** Prism.js или Highlight.js
- **Видео:** встроенный плеер + хранение ссылки (YouTube/VK Video) и/или локальный media upload

## 5. Роли и права доступа

### 5.1 Роли
- **Гость**
  - Просмотр публичных страниц: главная, курсы (ограниченно), новости, достижения, отзывы, контакты.
- **Пользователь (ученик/родитель)**
  - Регистрация и вход, просмотр доступных материалов, прохождение уроков и практики.
- **Преподаватель**
  - Создание и редактирование курсов/уроков по шаблону.
  - Добавление медиа, кода, практических задач, публикация/черновик.
  - Просмотр прогресса по своим курсам.
- **Администратор**
  - Полный доступ ко всем разделам.
  - Назначение ролей, удаление/блокировка пользователей, модерация отзывов, мониторинг.

### 5.2 Принципы управления ролями
- Регистрация: пользователь создается с базовой ролью `student`.
- Роли `teacher` и `admin` назначаются только администратором.
- Использовать Django Groups/Permissions + объектные проверки в представлениях.

## 6. Функциональные требования

### 6.1 Аутентификация и пользователи
1. Регистрация по **email + password**.
2. Вход/выход, смена и восстановление пароля.
3. Email как уникальный логин (кастомная модель User).
4. Профиль: ФИО, телефон (опционально), дата регистрации, роль, статус.
5. Администратор может:
   - просматривать список пользователей,
   - менять роль,
   - блокировать/разблокировать,
   - удалять пользователя.

### 6.2 Конструктор курсов (ключевое требование)
1. Преподаватель создает курс по **единому шаблону**:
   - название курса,
   - краткое и полное описание,
   - целевая возрастная группа,
   - уровень сложности,
   - обложка,
   - теги/направление (Python/Scratch/App Inventor/другое),
   - статус (черновик/опубликован).
2. Курс состоит из модулей и уроков (иерархия):
   - курс → модули → уроки.
3. Для урока преподаватель может добавить:
   - название,
   - теоретический контент (rich text),
   - изображения,
   - фрагменты кода с выбором языка и подсветкой синтаксиса,
   - видео (ссылка/загрузка),
   - практические задания,
   - дополнительные файлы.
4. Поддержать drag-and-drop сортировку модулей и уроков.
5. Поддержать предпросмотр урока до публикации.

### 6.3 Практика и проверка заданий
1. Для урока можно создавать задания типов:
   - тест (single/multiple choice),
   - ввод текста/числа,
   - кодовое задание (MVP: проверка по тест-кейсам для Python).
2. Проверка:
   - автоматическая проверка тестовых задач в backend,
   - для кодовых задач: запуск в изолированной среде (sandbox worker), лимиты по времени/памяти.
3. Сохранение результата:
   - статус выполнения,
   - баллы,
   - количество попыток,
   - время последней попытки.
4. В кабинете ученика: прогресс по курсу (%), список выполненных заданий.


### 6.4 Запись ученика на курс и учет прогресса
1. Ученик может присоединиться к курсу (enroll) через кнопку «Записаться/Присоединиться».
2. После записи курс появляется в личном кабинете ученика в разделе «Мои курсы».
3. Система фиксирует прогресс по каждому уроку и блоку:
   - **прочитано** (текстовые блоки/материалы),
   - **просмотрено** (видео-блоки),
   - **выполнено** (практические задания/тесты).
4. Материал считается «пройденным», если выполнены условия завершения урока (настраивается преподавателем):
   - базовый режим: открыты все блоки урока и выполнены обязательные задания;
   - расширенный режим: порог по баллам/проценту правильных ответов.
5. По курсу отображается агрегированный прогресс:
   - процент завершения (0–100%),
   - число пройденных/всего уроков,
   - дата последней активности.
6. В кабинете преподавателя отображается прогресс учеников по его курсам (в разрезе курса/модуля/урока).

### 6.5 Управление курсами (админ/преподаватель)
1. CRUD курсов, модулей, уроков и заданий.
2. Массовые действия: публикация/снятие, архивирование.
3. Фильтры и поиск по курсам.
4. Версионирование контента (минимум: история изменений ключевых полей).

### 6.6 Новости
1. CRUD новостей в админ-панели.
2. Поля: заголовок, анонс, полный текст, теги, дата публикации, изображение, статус.
3. Публичная выдача с пагинацией и фильтрами.

### 6.7 Достижения
1. CRUD достижений.
2. Поля: заголовок, описание, дата/год, категория, медиа, привязка к курсу (опционально).
3. Публичная страница достижений с фильтрацией.

### 6.8 Отзывы реальных людей
1. Форма отправки отзыва на сайте.
2. Поля: имя, тип автора (родитель/ученик/выпускник), текст, рейтинг, согласие на публикацию.
3. Обязательная модерация перед публикацией.
4. Отображение опубликованных отзывов в публичном разделе.

### 6.9 Контакты
1. Отдельная страница контактов.
2. Редактируемые поля: адрес, телефон, email, карта, ссылки на соцсети, график работы.

### 6.10 Мониторинг и администрирование
1. Endpoint `/health/`:
   - состояние приложения,
   - состояние БД,
   - timestamp.
2. Логирование ошибок backend.
3. Audit log действий администраторов и преподавателей (создание/правка/удаление контента).
4. Страница «Статус сервиса» в админке (uptime, активные пользователи, количество опубликованных курсов).

## 8. Архитектура приложения

### 8.1 Предлагаемые Django-приложения
- `accounts` — пользователи, роли, аутентификация
- `courses` — курсы, модули, уроки, прогресс
- `practice` — задания, попытки, автопроверка
- `news` — новости
- `achievements` — достижения
- `reviews` — отзывы и модерация
- `core` — главная, контакты, health endpoint
- `dashboard` — внутренняя панель управления
- `audit` — журнал действий

### 8.2 Ключевые сущности БД
- `User` (кастомный, email login)
- `Role`/`Group`
- `Course`
- `CourseModule`
- `Lesson`
- `LessonBlock` (текст/код/изображение/видео)
- `PracticeTask`
- `TaskAttempt`
- `Enrollment` (запись ученика на курс)
- `CourseProgress`
- `LessonProgressEvent` (прочитано/просмотрено/выполнено)
- `NewsPost`
- `Achievement`
- `Review`
- `ContactInfo`
- `AuditLog`

### 8.3 LessonBlock (унифицированный контент урока)
Для гибкости редактора уроков использовать блочную модель:
- `type`: `text`, `image`, `code`, `video`, `file`, `quiz_ref`, `task_ref`
- `order`: порядок блока
- `payload`: JSON с данными блока

Это позволит преподавателю собирать урок из готовых блоков в едином визуальном шаблоне.

1. Система запускается через `docker compose up`.
2. Регистрация/вход по email+password работают.
3. Администратор может назначать роли, удалять пользователей, управлять публикациями.

## 14. Риски и ограничения
1. **Автопроверка кода** требует безопасной изоляции выполнения (sandbox/worker).
2. **Видео-хостинг** при локальном хранении потребует место и контроль трафика.
3. **Сложность ролей** — необходима четкая матрица прав и тесты на доступ.
4. **Рост контента** — нужно планировать миграцию media в объектное хранилище.

## 16. Схема архитектуры (как это выглядит в целом)

### 16.1 Контур системы (высокоуровнево)
```text
[Пользователь: Гость/Ученик/Преподаватель/Админ]
                    |
                    v
              [Nginx Reverse Proxy]
                    |
                    v
             [Django Web (HTMX)]
             /        |          \
            /         |           \
           v          v            v
 [PostgreSQL]   [Media Storage]  [Redis]*
                                 |
                                 v
                           [Worker]*
```
`*` — опционально на MVP, но рекомендуется для проверки практических задач и фоновых процессов.

### 16.2 Логическая схема backend (Django apps)
```text
core  -> публичные страницы, health
accounts -> auth, роли, управление пользователями
courses -> курсы, модули, уроки, запись на курс
practice -> задачи, попытки, автопроверка
reviews -> отзывы и модерация
news -> новости
achievements -> достижения
dashboard -> внутренняя панель (admin/teacher)
audit -> журнал действий
```

### 16.3 Поток запроса (HTMX-сценарий)
```text
Браузер (HTMX click/submit)
    -> HTTP request
        -> Django view
            -> ORM query/update (PostgreSQL)
            -> partial template render
    <- HTML fragment response
Браузер обновляет только нужный блок страницы
```

### 16.4 Поток прогресса ученика (схематично)
```text
Ученик открывает урок/блок/видео/задачу
   -> событие прогресса (read/watch/do)
      -> LessonProgressEvent
      -> пересчет Lesson/CourseProgress
         -> обновление процента и статуса в "Мои курсы"
```

### 16.5 Схема сущностей (упрощенно)
```text
User (role)
  |--< Enrollment >-- Course --< CourseModule --< Lesson --< LessonBlock
  |                                           |
  |                                           |--< PracticeTask --< TaskAttempt
  |
  |--< CourseProgress
  |--< LessonProgressEvent

Course / User связаны через Enrollment
Прогресс считается из LessonProgressEvent + TaskAttempt
```

### 18.1 Принципы организации кода
1. **App-per-domain**: каждая предметная область в отдельном Django app.
2. **Разделение ответственности**:
   - `models.py` — данные,
   - `views.py`/`htmx_views.py` — HTTP-слой,
   - `services.py` — бизнес-логика,
   - `selectors.py` — сложные выборки чтения.
3. **Шаблоны и partials**:
   - полноразмерные страницы + отдельные HTMX-фрагменты.
4. **Статика**:
   - Alpine/HTMX инициализируются из `static/js`,
   - единая система стилей через `static/css`.
5. **Тесты рядом с приложениями + общий `tests/`** для интеграционных и e2e сценариев.
